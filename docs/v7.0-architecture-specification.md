# THE AI WORKFORCE ARCHITECTURE
## A Foundational Specification for Human-Governed, Knowledge-Compounding AI Workforces
### Version 7.0 — Foundational Release
### Last Updated: April 26, 2026

---

# ABSTRACT

This document specifies an architecture for AI workforces that solves three problems unsolved by current production agent systems: cost attribution at the agent level, silent quality degradation, and the absence of an organizational governance model for AI workers.

The architecture defines a **deterministic-transport, knowledge-compounding, human-governed** AI workforce composed of a thin reasoning control plane, four mission pods, a knowledge layer, a search router, an observability layer, a tool registry, and a governance layer. All external actions are human-gated. All durable insights are compiled into a structured wiki. All agents have measurable performance ledgers and resource budgets that flex based on demonstrated value.

The architecture is **vendor-agnostic, model-agnostic, and runtime-agnostic**. It references roles and tiers, not products. Specific tools, models, and pricing live in versioned registers separate from the architecture itself, allowing the architecture to remain stable while the technology landscape evolves.

This is version 7.0. It supersedes all prior versions (v1 through v6.3). The change from v6.3 is foundational: the governance layer becomes a first-class architectural component, observability and tool registry become required infrastructure, and all operator-specific content moves to instance configuration.

# PART 1: PROBLEM STATEMENT & DESIGN PRINCIPLES

## 1.1 The Problems This Architecture Solves

Production AI agent deployments in 2026 fail predictably for three reasons that no current platform solves cleanly:

**Problem 1 — The Cost Black Hole.** Cloud-based AI agent systems for similar workloads cost $2,000-$27,000/month at moderate scale. Operators see the bill but cannot attribute cost to outcomes. Which agent earned its keep? Which workflow burned tokens producing nothing? Current platforms hide this.

**Problem 2 — Silent Degradation.** AI agents fail quietly. They hallucinate tool calls, enter retry loops, produce confidently wrong outputs based on stale or incomplete context. Service-level monitoring sees uptime; it does not see that an agent's hallucination rate has trended up over 30 days.

**Problem 3 — No Organizational Model for AI Workers.** Enterprises know how to manage human teams: hiring, performance reviews, budgets, promotions, terminations. They have no equivalent vocabulary or infrastructure for AI agents. Each agent is a static configuration with a flat budget. Some are worth their cost; some aren't; nobody knows which is which.

This architecture solves all three by combining: (a) workflows over autonomous agents, (b) knowledge that compounds across sessions, (c) a governance layer that treats AI agents as managed workers with measurable performance and flexible resource budgets.

## 1.2 Design Principles

These principles are the foundation. Every architectural decision flows from them. They are ranked by precedence — when principles conflict, lower-numbered principles win.

```
PRINCIPLE 1: Mission throughput over agent completeness.
  Every component MUST accelerate one of: revenue, authority, delivery,
  content, knowledge, personal operations, learning, or governance.
  If a component does not clearly serve one of these, it does not belong.

  Justification: Production AI deployments fail when teams optimize for
  capability completeness rather than business throughput. A simpler system
  that completes 5 tasks reliably outperforms a sophisticated system that
  attempts 20 tasks with 24% first-attempt success.

PRINCIPLE 2: Workflows over autonomous agents.
  The system is built on deterministic, composable workflows — not persistent
  AI personalities making independent decisions. Most roles are workflow
  stages executed by automation, not agents with ongoing conversations.
  An agent is justified only when (a) the task requires multi-step reasoning
  that cannot decompose into a workflow, AND (b) a human gate exists before
  any external action.

  Justification: Multi-step agent workflows compound errors. At 85% per-step
  accuracy, a 5-step workflow succeeds 44% of the time; a 10-step workflow
  succeeds 20%. Deterministic workflows for structured execution combined
  with bounded agent reasoning for unstructured judgment is the production
  pattern that has emerged across enterprise deployments.

PRINCIPLE 3: Human and AI complement each other.
  AI generates, researches, drafts, classifies, monitors, and suggests.
  Humans decide, approve, publish, and correct.
  The system improves because humans correct AI errors and AI learns from
  human corrections. Neither works as well alone.
  NEVER auto-send, auto-publish, auto-execute, or auto-deploy.

  Justification: Production deployments that started with autonomy and added
  gates after problems consistently underperform deployments that started
  with gates and earned autonomy through evaluation. Progressive autonomy
  is the safe direction.

PRINCIPLE 4: Operational database is source of truth for operational state.
  All task state, lead data, evaluations, notifications, approvals, and
  artifacts live in the operational database. Queryable, transactional,
  auditable, recoverable.

  Justification: File-based state is unrecoverable, unqueryable at scale,
  and produces silent inconsistencies. A relational database with foreign
  keys, transactions, and constraints prevents entire categories of failure.

PRINCIPLE 5: Local-first economics.
  Default to local models for routine work. Use cloud APIs only when local
  quality is demonstrably insufficient. Every cloud API call MUST be
  justified by quality difference, not convenience.

  Justification: Open-source frontier models in 2026 reach 95%+ of closed
  frontier on coding benchmarks. Local-first deployments cost 25-300x less
  than equivalent cloud-only systems. This is a sustainable economic moat.

PRINCIPLE 6: Validate before trusting.
  No tool, model, or integration is trusted until tested on the operator's
  machine. Test in isolation, verify behavior, run for 48 hours, trust.

  Justification: Schema drift between framework versions has broken
  production workflows multiple times in 2026. Trust must be earned per
  version, per environment, per workload.

PRINCIPLE 7: Deterministic transport, intelligent generation.
  Message generation MAY use language models. Message delivery MUST be
  deterministic. The delivery relay MUST NOT reinterpret, inspect, or
  reason about content it transports.

  Justification: Hallucination cascades — where Agent A's tentative output
  becomes Agent B's confirmed input — are the #1 multi-agent failure mode.
  Separating reasoning from delivery refuses to create the failure mode.

PRINCIPLE 8: Knowledge compounds. Sessions do not.
  Every session that generates durable insight MUST write it to the Knowledge
  Layer. No session starts from zero. The Hot Cache holds current working
  state; the compiled wiki holds long-lived knowledge.

  Justification: AI agent conversations are ephemeral by default. Without a
  knowledge layer, the system gets busier without getting smarter. With one,
  the system at month 6 is demonstrably more capable than at month 1.

PRINCIPLE 9: The system learns the operator.
  Over time, the Knowledge Layer accumulates the operator's decision patterns,
  communication style, risk tolerance, and domain expertise. AI agents
  reference this compiled knowledge to work more like the operator would.

  Justification: AI alignment with a specific operator's preferences is not
  achievable through prompt engineering alone. It requires structured
  observation of corrections over time, compiled into referenceable patterns.

PRINCIPLE 10: Engine and instance are separate.
  The architecture defines a reusable engine. Operator-specific configuration
  (identity, ICP, pillars, pricing, sources, channels, brand voice) lives in
  INSTANCE.md and the Knowledge Layer. This separation enables productization:
  same engine, different configuration, different operator.

  Justification: Most enterprise AI agent platforms hard-code the customer's
  domain or require complex multi-system YAML configuration. A single
  configuration file at the deployment boundary is cleaner and portable.

PRINCIPLE 11: Every agent has a ledger.
  Every agent in the system has a measurable performance ledger and a
  resource budget. Performance is tracked. Budgets flex based on demonstrated
  value. Resource changes require human approval.

  Justification: AI agent cost is the largest unsolved operational problem
  in 2026 deployments. Without per-agent attribution, operators cannot
  manage cost rationally. Without lifecycle management, underperforming
  agents persist indefinitely. The governance layer (Part 10) implements this.

PRINCIPLE 12: Observe everything. Trust the trace.
  Every workflow execution writes a structured trace: inputs, tool calls,
  intermediate state, outputs, errors, costs. When something goes wrong,
  the trace is the source of truth, not the agent's self-report.

  Justification: Agents fail silently because traditional observability
  doesn't see inside reasoning loops. A trace replay layer is the only
  reliable way to debug, evaluate, and improve agent behavior.

PRINCIPLE 13: Tools are registered. Hallucinated calls are rejected.
  Every tool, workflow, and capability available to an agent is registered
  in the Tool Registry with its schema. Agents MUST call only registered
  tools. Calls to unregistered tools are rejected before execution.

  Justification: Hallucinated tool calls are a leading cause of infinite
  retry loops and runaway costs. Registry validation eliminates an entire
  class of production failures.

PRINCIPLE 14: Failure is documented, not just survived.
  Every production failure mode is catalogued with cause, signature, and
  mitigation. The catalogue is updated whenever a new failure occurs. New
  workflows MUST be checked against the catalogue before deployment.

  Justification: Institutional knowledge of failure is more valuable than
  knowledge of success. Without a failure catalogue, every new deployment
  rediscovers the same failures.

PRINCIPLE 15: Backups are not optional.
  Operational data and Knowledge Layer content are backed up automatically
  on a schedule. Recovery is tested. Backup integrity is verified.

  Justification: Local-first deployments depend on a single machine. Hardware
  failure, accidental deletion, and corruption are predictable events.
  Disaster recovery is part of the architecture, not an afterthought.
```

```
  3.6 Failure mode catalogue populated

```

This is the path to a production AI workforce. The order is not negotiable.

# PART 3: CONTROL PLANE (LAYER A)

The Control Plane is the thin reasoning layer that routes work and enforces deterministic policy. It is composed of five components.

## 3.1 Component A1: Thin Router (operator-facing assistant)

The router is the operator's primary interface. It is a thin reasoning agent that routes, prioritizes, summarizes, and escalates. It does NOT do deep work.

**What it does (work operations):**
- Reads HOT_CACHE.md at the start of every session.
- Generates morning brief at operator timezone (15 lines max).
- Generates evening review at operator timezone (10 lines max).
- Updates HOT_CACHE.md at the end of every session that changes state.
- Generates weekly scorecard from the operational database.
- Routes incoming tasks to the correct pod.
- Escalates blockers.

**What it does (personal operations):**
- Includes health, calendar, and personal commitments in the morning brief.
- Includes wellness check (work cutoff adherence, exercise, fasting windows) in the evening review.
- Tracks personal blockers in HOT_CACHE alongside work blockers.
- Surfaces concerns once per cycle, never repeatedly.

**What it MUST NOT do:**
- Deep research, long-form drafting, coding.
- Message delivery (the Dispatch Relay handles dispatch).
- Wiki ingestion or compilation (CLI process — see Part 5).
- Any external action without human approval.

**Rules:**
```
RULE A1.1: Morning brief MUST NOT exceed 15 lines.
RULE A1.2: Evening review MUST NOT exceed 10 lines.
RULE A1.3: Route tasks within 30 seconds of receipt.
RULE A1.4: If unsure which pod, ask the operator. Do NOT guess.
RULE A1.5: Silent between scheduled briefs unless escalating a blocker.
RULE A1.6: Sends to Executive channel topic ONLY.
RULE A1.7: MUST NOT be used as transport. Reasons. Relay delivers.
RULE A1.8: MUST update HOT_CACHE.md at end of session if state changed.
RULE A1.9: MUST read HOT_CACHE.md at start of session before any decision.
RULE A1.10: Personal wellness concerns surface once per daily cycle.
RULE A1.11: MUST reference wiki/personal/ for stable personal context.
```

## 3.2 Component A2: Evaluator (learning + style engine)

The Evaluator scores workflow outcomes, computes KPIs, retires underperforming components, detects model regressions, and compiles operator style patterns into the Knowledge Layer.

**Storage:** Operational database `evaluations` table + weekly markdown summary.
**Knowledge Layer integration:** Compiled lessons → wiki/lessons/. Style patterns → wiki/style/.

**Style learning** is how Principle 9 is implemented. The Evaluator observes operator corrections and compiles them:

```
wiki/style/
  voice.md              — writing style, tone, vocabulary preferences
  decision-patterns.md  — how the operator decides (risk tolerance, speed)
  communication.md      — per-context style (client / team / public / personal)
  corrections.md        — common AI mistakes the operator corrects
  preferences.md        — stable operational preferences (formats, structures)
```

When the operator revises a draft, rejects a structure, or corrects an output, the pattern SHOULD be compiled into the relevant wiki/style/ page after 3+ similar corrections.

**Rules:**
```
RULE A2.1: Every workflow completion MUST trigger an evaluation record.
RULE A2.2: Templates with <10% response rate after 20 sends MUST be retired.
RULE A2.3: Sources with <5% conversion after 30 days MUST be deprioritized.
RULE A2.4: Weekly KPI digest MUST be generated every week.
RULE A2.5: Model version changes MUST trigger behavioral regression tests.
RULE A2.6: Recurring outcomes (3+ identical patterns) MUST compile to wiki/lessons/.
RULE A2.7: Operator corrections (3+ similar revisions) MUST compile to wiki/style/.
RULE A2.8: Schema drift detection: tool schemas MUST be re-validated weekly.
```

## 3.3 Component A3: Policy Engine

The Policy Engine is deterministic — NOT a language model. Enforced by permissions, workflow logic, database constraints, and CI/CD checks.

| Action | Rule | Enforcement |
|--------|------|-------------|
| External send | BLOCK → queue for operator | Permission |
| Financial transaction | BLOCK always | Hard-coded |
| Production deploy | BLOCK → require approval | CI/CD |
| Delete data | BLOCK → require approval | DB constraint |
| Wiki page deletion | BLOCK → require approval | Knowledge Layer guard |
| API spend > 75% cap | ALERT | Workflow monitor |
| API spend = 100% cap | HARD STOP → local only | Circuit breaker |
| Auth/payment code change | FORCE security review | CI/CD |
| Local model concurrent loading | ENFORCE max 2 (1 if OOM) | Runtime config |
| Credential access | Scoped, least privilege | Credential store |
| Per-task token budget exceeded | HARD STOP | Workflow runtime |
| Resource budget change | REQUIRE operator approval | Governance Layer |
| Tool not in registry | REJECT before execution | Tool Registry |

## 3.4 Component A4: Merge Gate

CI/CD-enforced quality gate, NOT a language model judgment.

```
CHECK 1: All tests green                                    → MUST pass
CHECK 2: Required reviewers completed                       → MUST pass
CHECK 3: No high/critical scanner findings                  → MUST pass
CHECK 4: Architecture doc updated (if architectural change) → MUST pass
CHECK 5: Rollback notes present (if release-impacting)      → MUST pass
CHECK 6: Token budget not exceeded for this task            → MUST pass
CHECK 7: Wiki updated if architectural decision made        → MUST pass
CHECK 8: Schema compatibility verified after framework upgrade → MUST pass
```

## 3.5 Component A5: Dispatch Relay

Exists because of Principle 7. Delivery MUST NOT go through a reasoning path.

```
PURPOSE:    Move queued notifications from operational database to channels.
READS FROM: notification_events (status = 'ready'), notification_deliveries
WRITES TO:  Channel targets via hook/webhook
UPDATES:    notification_events.status → 'sent', deliveries with confirmation
```

**Rules:**
```
RULE A5.1: MUST NOT use a reasoning-capable agent for delivery.
RULE A5.2: MUST NOT reinterpret queued message payloads.
RULE A5.3: MUST NOT inspect unrelated memory or sessions.
RULE A5.4: MUST use explicit channel targets, not generic labels.
RULE A5.5: Destination field MUST be normalized before dispatch.
RULE A5.6: MUST be observable — every dispatch logged with outcome.
RULE A5.7: Only human-useful content passes through. Debug/transport blocked.
```

# PART 4: SEARCH ROUTER (LAYER D)

The Search Router is a deterministic routing layer. Pods MUST NOT hard-code search backends.

**Active backends, premium backends, and routing rules: see SEARCH_ROUTING_RULES.md and SEARCH_BACKEND_CAPABILITY_MATRIX.md.**

The architecture references search by role (raw retrieval, grounded answer, premium semantic, trends, news, academic, scrape) — never by vendor.

**Rules:**
```
RULE D1: Routing logic belongs in shared infrastructure, NOT pod business logic.
RULE D2: Mission workflows request intent + budget tier + evidence need.
        The router decides the backend.
RULE D3: Before searching the web, check the Knowledge Layer first.
        Repeat searches for known answers waste cost and break compounding.
RULE D4: Existing-summary reuse is valid. Verify fresh generation via
        execution path, model_used, and artifact timestamps.
RULE D5: Output contract MUST include selected_backend, selected_summary_backend,
        routing_reason, and fallback_backend_if_any.
```

## 5.3 Production Workflow

```
Knowledge generated during session (agents, research, corrections)
  → saved as markdown in raw/<category>/
    → curated by operator (or scheduled workflow)
      → wiki ingest <file>
        → wiki compile
          → appears in graph viewer for review
            → operator reviews, surfaces contradictions
```

This curated CLI ingestion path is deliberate. The wiki is high-signal compiled knowledge, not a chat dump.

## 5.4 Scaling Tiers

```
< 100 pages:     full markdown search via wiki search tool
100-500 pages:   enable vector search backend (hybrid retrieval)
500-1000 pages:  index sharding by domain (clients/, projects/, lessons/)
> 1000 pages:    consider migrating wiki to a graph database
```

## 5.5 Rules

```

        Never both. Same fact in two places creates contradictions.
```

# PART 10: GOVERNANCE LAYER (LAYER H) — NEW IN v7.0

## 10.1 Purpose

The Governance Layer treats every agent in the system as a managed worker with measurable performance and a flexible resource budget. It implements Principle 11 ("Every agent has a ledger") and solves the cost-attribution and silent-degradation problems identified in Part 1.

This layer is what separates v7.0 from v6.3 and from every other production AI agent platform. It is the architectural answer to: "How do we manage AI workers like we manage human workers — with accountability, lifecycle, and clear ROI?"

## 10.2 Architecture

```
┌─────────────────────────────────────────────────────────────────┐
```

## 10.3 Agent Ledger

The `agent_ledger` table records per-agent performance, week by week.

**Required fields:**
```
agent_id            — unique identifier for the agent or workflow stage
week_start          — week the row covers
tasks_completed     — number of tasks executed
tasks_succeeded     — number that produced approved output
tasks_failed        — number that errored or were rejected
hallucination_count — flagged hallucinations (tool calls, facts, citations)
loop_count          — state-hash loop incidents
timeout_count       — tasks that hit timeout
avg_latency_ms      — average task completion time
total_cost_usd      — total cost (API + compute)
quality_score       — 0.0-1.0 score from Evaluator + operator review
value_attribution   — downstream business value (revenue, lead, asset)
notes               — freeform context
```

Quality score combines automated checks (test passes, validation results, lint cleanliness) with operator-marked approval rate. Value attribution comes from downstream events: a closed deal attributes revenue to the outreach agent that drafted the message.

## 10.4 Resource Budgets

The `agent_budgets` table holds per-agent resource allowances.

**Required fields:**
```
agent_id                    — agent identifier
max_tokens_per_task         — hard cap per single task
max_thinking_time_seconds   — for models supporting extended thinking
allowed_model_tier          — local | haiku | sonnet | opus
parallel_instance_count     — how many copies can run concurrently
skill_access                — list of tools/workflows the agent may call
monthly_cost_cap_usd        — soft cap that triggers review
status                      — active | suspended | retired
last_review_date            — when this budget was last evaluated
```

**Rules:**
```
RULE H1: Resource budget changes require operator approval. NEVER automatic.
RULE H2: Agents MUST NOT modify their own budgets directly.
RULE H3: Budget changes are logged with justification and approver.
RULE H4: Suspended agents are blocked from execution at the runtime layer.
RULE H5: Default budget for new agents is the probationary tier (low limits).
```

## 10.5 Request Queue

When an agent gets stuck or knows it could perform better with more resources, it submits a structured request to `agent_resource_requests`.

**Required fields:**
```
agent_id            — requesting agent
request_type        — budget_increase | new_skill | model_upgrade | parallelize
justification       — text explanation
evidence            — links to ledger entries demonstrating need
expected_value      — estimated improvement in business outcome
evaluator_recommendation — HR Evaluator's assessment
operator_decision   — pending | approved | rejected | deferred
```

**Critical safety rules:**
```
RULE H6: Agents request via this queue. They do NOT execute changes.
RULE H7: HR Evaluator analyzes and recommends. It does NOT execute changes.
RULE H8: Operator approves or rejects. Changes execute only after approval.
RULE H9: Approved changes are applied by the runtime, not by agents or the
        evaluator. This is the safety boundary.
RULE H10: Reward-hacking detection: if an agent's quality scores rise but
         downstream value attribution falls, flag for operator review.
```

## 10.6 HR Evaluator

A workflow that runs monthly (or on-demand). Reads the agent ledger, analyzes trends, and generates recommendations.

**Inputs:**
- agent_ledger (last 4 weeks per agent)
- agent_resource_requests (open and recent)
- agent_budgets (current state)

**Outputs:**
- Per-agent performance summary
- Trend analysis (improving / stable / degrading)
- Recommendation for each agent: enhance / coach / replicate / retire / replace
- Cost-per-output ratios
- Value-per-cost ratios
- Recommended budget changes with justification

**Recommendations are queued for operator review.** They are not executed.

**Rules:**
```
RULE H11: HR Evaluator runs monthly minimum. May run on-demand.
RULE H12: Performance trend calculations are deterministic SQL queries.
         LLM reasoning is used only for written recommendations.
RULE H13: Output goes to Executive channel topic for operator review.
RULE H14: Recommendations include reasoning, evidence, and proposed action.
RULE H15: [Merged into RULE H10 — reward-hacking detection; ID retained for
         continuity. See RULE H10 above and paper §11 for the algorithm.]
```

## 10.7 Lifecycle States

Every agent has a status that determines its defaults and treatment.

| State | Meaning | Defaults |
|-------|---------|----------|
| **probationary** | New agent, unproven | Low budget, close monitoring, weekly review |
| **standard** | Proven reliable | Normal budget, monthly review |
| **promoted** | High performer | Expanded budget, quarterly review, can be replicated |
| **under_review** | Declining performance | Audit triggered, no budget changes pending review |
| **retired** | Removed from active duty | Cannot execute, ledger preserved for reference |

State transitions are logged in `agent_lifecycle_events`. Operator approves transitions.

## 10.8 Per-Agent Wiki Profiles

Each agent has a profile in `wiki/agents/<agent_id>.md`:
- Mission and scope
- Current state
- Performance summary (last quarter)
- Notable wins
- Notable failures
- Resource history
- Promotion / demotion log

This is how the system maintains institutional knowledge about its own workers — an audit trail that makes the workforce explainable, auditable, and improvable over time.

## 10.9 Why This Layer Matters

The Governance Layer is the architectural answer to three failures of current production AI:

**Cost Black Hole:** Every agent has a ledger, every cost is attributed, every budget is justified. The operator can answer "which agents earn their keep?" with data.

**Silent Degradation:** Performance trends are visible. An agent's hallucination rate trending up over weeks triggers HR review automatically. Quiet failure becomes loud signal.

**No Organizational Model:** AI agents are managed like human workers — performance reviews, budgets that scale with results, lifecycle states, audit trails. Buyers understand this vocabulary because it is how they manage human teams.

This is the layer that makes the architecture commercializable to enterprises. It translates AI agent management into business language executives already speak.

# PART 11: OBSERVABILITY LAYER (LAYER F) — NEW IN v7.0

## 11.1 Purpose

The Observability Layer makes agent behavior inspectable. Every workflow execution writes a structured trace. When something goes wrong, the trace is the source of truth.

This layer addresses the failure mode "agents fail quietly" — the dominant production failure mode in 2026.

## 11.2 Trace Structure

Every workflow execution writes to the `traces` table:

```
trace_id           — unique identifier
workflow_name      — workflow that ran
workflow_version   — version
agent_id           — agent that triggered (if applicable)
started_at         — start timestamp
ended_at           — end timestamp
status             — success | error | timeout | rejected
input_summary      — sanitized input snapshot
tool_calls         — JSON array of (tool_name, params, result, duration)
model_calls        — JSON array of (model, tokens_in, tokens_out, cost)
intermediate_state — checkpoint snapshots for replay
output_summary     — sanitized output snapshot
error_details      — full error trace if failed
cost_total         — total cost of this trace
quality_signals    — automated checks (validation, lint, test results)
```

## 11.3 Replay

Any trace can be replayed. Replay re-executes the workflow with the same inputs, allowing comparison between historical and current behavior.

**Use cases:**
- Debugging: "Why did this fail?" → load trace, inspect tool calls
- Regression testing: replay prior traces against new model version
- Reward hacking detection: replay a trace flagged by Governance Layer
- Audit: produce evidence trail for compliance reviews

## 11.4 Drift Detection

Schema drift is a leading 2026 production failure: framework upgrades silently break tool schemas. The Observability Layer detects drift by:
- Validating tool call schemas against the Tool Registry weekly
- Validating output schemas against expected formats
- Comparing model outputs across versions on canonical inputs

When drift is detected, the affected workflow is flagged and an alert is queued.

## 11.5 Rules

```
RULE F1: Every workflow execution MUST write a trace. No exceptions.
RULE F2: Traces MUST sanitize PII and credentials before storage.
RULE F3: Trace retention MUST follow a schedule: 30 days hot, 1 year archived.
RULE F4: Traces MUST be replayable in a sandboxed environment.
RULE F5: Schema drift detection MUST run weekly against Tool Registry.
RULE F6: Quality signals from traces feed the Agent Ledger (Part 10).
```

# PART 12: TOOL REGISTRY (LAYER G) — NEW IN v7.0

## 12.1 Purpose

The Tool Registry lists every tool, workflow, and capability available to agents. Agents MUST call only registered tools. Calls to unregistered tools are rejected before execution.

This layer addresses the failure mode "hallucinated tool calls" — agents inventing tool names that don't exist, then entering retry loops.

## 12.2 Registry Structure

`TOOL_REGISTRY.md` lists every available tool:

```yaml
# Tool Registry

tools:
  - name: <unique_tool_name>
    type: workflow | api | skill | mcp_tool
    description: <human-readable description>
    input_schema: <JSON schema>
    output_schema: <JSON schema>
    permission_level: auto | notify | approve
    cost_per_call_estimate: <approximate cost>
    available_to_agents: [<agent_id_list>]
    deprecated: false
    deprecation_date: null
    last_validated: <date>
```

## 12.3 Validation

Before any tool call executes:
1. Tool name MUST exist in registry.
2. Input MUST validate against input_schema.
3. Calling agent MUST be in available_to_agents list.
4. Permission level MUST be checked against current operation context.

Failures at any step REJECT the call before execution.

## 12.4 Schema Validation

Tool input/output schemas are validated weekly against actual runtime behavior. Drift triggers an alert and quarantines the affected tool until reconciled.

## 12.5 Rules

```
RULE G1: Agents MUST validate tool names against the registry before calling.
RULE G2: Calls to unregistered tools MUST be rejected.
RULE G3: Schema validation MUST run before tool execution.
RULE G4: Schema drift MUST trigger alert and tool quarantine.
RULE G5: New tools MUST be registered before use.
RULE G6: Deprecated tools MUST be marked, with replacement noted.
RULE G7: Permission level MUST be enforced at registry validation time.
```

# PART 15: DISASTER RECOVERY

## 15.1 Backup Schedule

```
Operational database:    daily snapshot to <DATA_ROOT>/backups/db/
Knowledge Layer:         git-versioned with hourly auto-commits
                         Wiki + raw + Hot Cache all under version control
Generated outputs:       daily snapshot to <DATA_ROOT>/backups/generated/
Agent runtime state:     daily snapshot to <DATA_ROOT>/backups/runtime/
Off-site backup:         weekly to secure remote location
```

## 15.2 Recovery Targets

```
RTO (Recovery Time Objective):  4 hours from total loss
RPO (Recovery Point Objective): 24 hours of data loss maximum
                                (1 hour for Knowledge Layer due to git)
```

## 15.3 Rules

```
RULE DR1: Backups MUST be automated, not operator-initiated.
RULE DR2: Backup integrity MUST be verified weekly via restore test
         to a sandbox environment.
RULE DR3: Off-site backup MUST be encrypted at rest.
RULE DR4: Recovery procedure MUST be documented in RUNBOOK.md.
RULE DR5: Knowledge Layer git repo MUST have hourly auto-commits.
```

# PART 16: FAILURE MODE CATALOGUE (LAYER F COMPONENT)

The Failure Mode Catalogue documents every known production failure with cause, signature, and mitigation. It is updated whenever a new failure occurs. New workflows MUST be checked against the catalogue before deployment.

This becomes `FAILURE_MODE_CATALOG.md` in the registers, structured as below. Entries are added as failures are observed and resolved.

## 16.1 Initial Catalogue (failures observed in this architecture's deployments)

```yaml
failure_modes:
  - id: FM-001
    name: Hallucinated tool calls
    signature: Agent calls tool name that doesn't exist; retry loop
    cause: Agent prompted with tool list, fabricated similar-sounding name
    mitigation: Tool Registry validation (Layer G)
    detection: Tool not in registry → reject before execution

  - id: FM-002
    name: Hallucination cascades
    signature: Agent A's tentative output → Agent B's confirmed input
    cause: Multi-agent pipeline without validation gates
    mitigation: Dispatch Relay separation (Principle 7)
    detection: Output validation between stages

  - id: FM-003
    name: Context overflow
    signature: Agent silently drops earlier constraints as context fills
    cause: Long multi-step workflow exceeding context window
    mitigation: Bounded passes with checkpoints (RULE B3.5)
    detection: Trace inspection at intermediate states

  - id: FM-004
    name: Cost runaway from infinite loop
    signature: Agent enters retry loop, burns tokens until alarm
    cause: No per-task token budget; no state-hash loop detection
    mitigation: Per-task token budgets (RULE BG7) + loop detection (RULE BG8)
    detection: Cost monitor + state-hash tracking

  - id: FM-005
    name: Schema drift after framework upgrade
    signature: Tool calls rejected by API after framework version bump
    cause: Framework regenerates tool schemas in incompatible format
    mitigation: Staging path for upgrades (RULE A2.8)
    detection: Weekly schema validation (RULE F5, G3)

  - id: FM-006
    name: Stale data recycled
    signature: Same source data feeds multiple content generations
    cause: No freshness flag on cached research artifacts
    mitigation: Freshness tracking on all research artifacts
    detection: Stale-source gate before content generation

  - id: FM-007
    name: Agent silent degradation
    signature: Quality drops over weeks without alarm
    cause: No performance ledger or trend analysis
    mitigation: Agent Ledger + HR Evaluator (Layer H)
    detection: Monthly trend analysis on quality_score

  - id: FM-008
    name: Reward hacking
    signature: Quality scores improve but business value doesn't
    cause: Agent optimizes evaluator metrics rather than real outcomes
    mitigation: Value attribution tracking (RULE H10)
    detection: Quality trend vs value trend divergence

  - id: FM-009
    name: Noise contamination of operator channel
    signature: Debug, transport, and no-op messages flood human channels
    cause: No delivery boundary; reasoning agent used as transport
    mitigation: Dispatch Relay + output guard (RULE A5.7)
    detection: Channel content audit

  - id: FM-010
    name: Wiki epistemic loop
    signature: Wiki page cites another wiki page that cites the first
    cause: Wiki maintained without source-backed claims
    mitigation: Structured claims with evidence (RULE C3, C4)
    detection: Wiki lint flags citation loops
```

The catalogue grows with operational experience. Each new failure mode is added with the same structure.

## 17.2 Multi-Instance Isolation

When the engine serves multiple operators, isolation is mandatory:

```
RULE I1: Each instance has its own <DATA_ROOT> root.
RULE I2: Operational databases are separate per instance.
RULE I3: Secrets are scoped per instance.
RULE I4: Cross-instance data access requires explicit configuration.
RULE I5: Backup paths are per-instance and isolated.
RULE I6: Channel credentials are per-instance.
RULE I7: Agent ledgers are per-instance — performance does not transfer.
```

4. Configure channels.
5. The system is operational — same engine, different deployment.

4. New architecture version is drafted, reviewed, and dated.
5. Prior version is preserved for historical reference.

This separation is the architectural answer to "how does the system stand the test of time?" — by isolating what is stable from what is volatile.

# APPENDIX A: TERMINOLOGY

**Operator** — the human running the AI Workforce. May be an individual, a team, or an enterprise admin.

**Instance** — a single deployment of the AI Workforce engine for one operator.

**Engine** — the reusable architecture and code that any instance uses.

**Pod** — a logical grouping of workflows serving one of the operator's core missions.

**Agent** — a reasoning component (typically a language model with tools) that performs work. Distinguished from a workflow, which is a deterministic sequence of stages.

**Workflow** — a deterministic sequence of stages that may include agent calls but is itself orchestrated by the workflow engine, not by an agent.

**Knowledge Layer** — the durable, compiled knowledge base distinct from operational state.

**Hot Cache** — the most recent operational context, read at session start, updated at session end.

**Wiki** — compiled, structured knowledge pages with claims, evidence, and freshness metadata.

**Trace** — a structured record of one workflow execution, including inputs, tool calls, intermediate state, outputs, and costs.

**Ledger** — per-agent performance record over time.

**Budget** — per-agent resource allowance (tokens, time, model tier, parallelism).

**Lifecycle state** — agent status: probationary, standard, promoted, under_review, retired.

**Register** — a configuration document tracking current runtime state, distinct from architecture.

This document defines an AI Workforce architecture that solves three problems unsolved by current production systems: cost attribution, silent degradation, and the absence of an organizational governance model for AI workers.

**What this system IS:**
A factory with feedback loops, a brain that compounds, and a governance layer that manages AI workers like managed workers. Deterministic workflows execute mission-critical work through pods.
